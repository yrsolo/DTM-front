## Что решает проект

`Спонсорские ТНТ` — это read-only frontend для просмотра и навигации по задачам производства. Его задача не редактировать исходные данные, а давать рабочий визуальный слой поверх snapshot-данных:

- кто за что отвечает;
- какие задачи активны сейчас;
- какие milestones приближаются;
- как выглядит общая картина по задачам и по дизайнерам.

Проект ориентирован на ежедневное использование: быстрый вход, плотный timeline, мгновенное открытие карточки задачи, сохранение пользовательского состояния в браузере и возможность тонко настроить визуальный слой без пересборки.

## Источник данных

Frontend не является источником правды по задачам.

Источники данных:
- локальный snapshot JSON;
- demo snapshot;
- API-контур, который отдаёт frontend-ориентированный snapshot.

Backend, таблицы, процессы синхронизации и административная логика находятся вне этого репозитория.

## Текущая аудитория

- дизайнеры, которым нужен оперативный обзор своих задач;
- лиды и менеджеры, которым нужен обзор по задачам и исполнителям;
- владелец продукта и дизайнер интерфейса, которые настраивают внешний вид через workbench.

## Что умеет продукт сейчас

### 1. Страница `Задачи`

Главный рабочий экран:
- плотный таймлайн;
- левый блок со строками;
- переключение группировки;
- zoom и прокрутка;
- фильтры статусов и окна дат;
- tooltip по задаче;
- карточка задачи с календарём и milestones.

### 2. Страница `Дизайнеры`

Альтернативный обзор:
- колонки по дизайнерам;
- карточки задач внутри колонок;
- hover-tooltip с milestones, менеджером и history;
- цветовая подкраска карточек от task palette.

### 3. Runtime-тюнинг интерфейса

Встроенный workbench даёт:
- настройку material и цветовых слоёв;
- настройку таймлайна, milestones, drawer и panels;
- настройку runtime defaults;
- импорт, экспорт, локальное сохранение и загрузку deploy preset.

## Что сознательно не входит в scope

- редактирование задач;
- изменение milestone-данных из интерфейса;
- backend-логика;
- auth/admin workflows внутри этого репозитория;
- управление исходными таблицами.

## Текущее состояние зрелости

Проект находится в рабочем прикладном состоянии: интерфейс уже используется как операционный обзор задач, при этом остаётся насыщенным runtime-настройками и продолжает эволюционировать в коде. Поэтому документация должна опираться на текущую реализацию, а не на старые MVP-планы.
