# Пользовательские сценарии

Этот документ описывает пользовательские сценарии без углубления во внутреннюю реализацию.

Для кого:
- PM;
- владелец продукта;
- дизайнер интерфейса;
- инженер, которому важно понять пользовательский путь до чтения архитектуры.

## Сценарий: работа со страницей `Задачи`

Цель:
- быстро увидеть общую картину по активным задачам и срокам.

Шаги:
1. Пользователь открывает приложение.
2. Приложение восстанавливает сохранённый режим страницы или показывает `Задачи` по умолчанию.
3. На экране отображаются строки, таймлайн, pinned-шкала дат, dock управления масштабом и фильтрами, кнопки группировки.
4. Пользователь меняет группировку, масштаб или фильтры и получает тот же timeline в другом срезе.

Результат:
- пользователь видит сжатый, но полный обзор задач и их milestones.

## Сценарий: работа со страницей `Дизайнеры`

Цель:
- посмотреть задачи в разрезе исполнителей.

Шаги:
1. Пользователь переключается на страницу `Дизайнеры` в правом верхнем блоке.
2. Интерфейс строит адаптивное число колонок по дизайнерам, присутствующим в данных.
3. Внутри колонок отображаются карточки задач с брендом, форматом и шоу.
4. При наведении показывается tooltip с milestones, менеджером и history.

Результат:
- пользователь видит загрузку дизайнеров в карточном виде без таймлайна.

## Сценарий: открытие карточки задачи

Цель:
- получить подробности по конкретной задаче.

Шаги:
1. Пользователь кликает по строке заголовка или по самой диаграмме задачи.
2. Открывается `TaskDetailsDrawer`.
3. Внутри отображаются метаданные, badges, список milestones, календарь и history.
4. При включённой анимации открытие и закрытие происходят плавно.

Результат:
- пользователь получает подробный контекст без ухода со страницы.

## Сценарий: работа с фильтрами, масштабом и группировкой

Цель:
- быстро менять срез представления без потери позиции в данных.

Шаги:
1. Пользователь меняет zoom кнопками или `Alt + колесо`.
2. Пользователь меняет группировку через mode panel.
3. Пользователь включает или выключает статусы и другие runtime-фильтры.
4. Приложение перестраивает timeline или board, сохраняя состояние в браузере.

Результат:
- пользователь получает нужный операционный ракурс без смены данных.

## Сценарий: использование workbench

Цель:
- настроить визуальную систему и runtime defaults без пересборки.

Шаги:
1. Пользователь открывает workbench.
2. Переходит по каноническим вкладкам: `Foundation`, `Surfaces`, `Timeline`, `Tasks Table`, `Drawer`, `Milestones`, `Motion`, `Workbench`, `Defaults`.
3. Меняет параметры через range-контролы и бинарные кнопки.
4. Сохраняет локально, экспортирует JSON или загружает deploy preset.

Результат:
- визуальная система и runtime defaults меняются прямо в браузере.

## Сценарий: загрузка данных

Цель:
- получить snapshot из нужного источника.

Шаги:
1. Приложение загружает runtime-конфиг.
2. Пользователь может оставить локальный режим, demo mode или включить test API.
3. `useSnapshot` либо читает local snapshot, либо делает запрос в API.
4. Ответ нормализуется и рендерится в UI.
5. Последний успешный snapshot и meta кэшируются в localStorage.

Результат:
- пользователь продолжает работать даже при временных проблемах сети, если в браузере уже есть валидный кэш.
